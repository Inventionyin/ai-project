我先用现有代码作为基线约束：

- 统一响应壳： {code,message,data,requestId} ， common.py
- 全局异常仍返回 HTTP 200，业务错误写在 code/message ， main.py
- 现有执行状态不变： QUEUED/RUNNING/PASSED/FAILED/CANCELED ， enums.py
- 现有运行幂等参考 Idempotency-Key 机制， runs.py 、 run.py
目标范围

- 保留现有套件执行链路。
- 新增“测试用例界面直连挂载”能力：测试数据挂载 + 接口挂载 + 批量执行。
- 新接口统一走 /api 前缀。
统一协议

- 鉴权头
  
  - Authorization: Bearer <token> （沿用现状， deps.py ）
- 请求追踪头
  
  - 可传： X-Request-Id （不传则后端生成， deps.py ）
- 幂等头（仅 POST 创建类）
  
  - Idempotency-Key: <1..128>
- 响应统一结构
```
{
  "code": 0,
  "message": "ok",
  "data": {},
  "requestId": "req_a1b2c3d4e5f6"
}
```
- 分页统一结构
```
{
  "code": 0,
  "message": "ok",
  "data": {
    "page": 1,
    "pageSize": 20,
    "total": 123,
    "items": []
  },
  "requestId": "req_a1b2c3d4e5f6"
}
```
字段规范

- 通用 ID 字段
  
  - string ，长度 1..64 （与现有 IdStr 一致， types.py ）
- 时间字段
  
  - UnixTs （秒级整型， >=0 ， types.py ）
- 枚举字段
  
  - triggerType : MANUAL|CRON|CI|WEBHOOK
  - runStatus : QUEUED|RUNNING|PASSED|FAILED|CANCELED
  - caseRunStatus : QUEUED|RUNNING|PASSED|FAILED|SKIPPED
- JSON 扩展字段
  
  - params : object ，KV 结构，建议限制最大 2000 keys，单值最大 4KB
  - headers : object<string,string>
  - authRef : object （只存引用，不存明文密钥）

错误码约定（兼容现有）

- 现有全局映射继续保留（来源于异常状态码映射）： main.py
- 建议在 message 使用稳定错误标识，便于前端国际化和分支处理。
- 错误码使用规则
  
  - 0 : 成功
  - 40001 : 参数错误/业务前置条件不满足
  - 40101 : 未认证
  - 40301 : 无权限
  - 40401 : 资源不存在
  - 40901 : 并发冲突/幂等冲突
  - 42901 : 限流
  - 50301 : 下游或数据库不可用
  - 50001 : 未知内部错误
- 典型错误样例
```
{
  "code": 40901,
  "message": "idempotency_key_payload_mismatch",
  "data": {
    "idempotencyKey": "abc-001"
  },
  "requestId": "req_xxx"
}
```
```
{
  "code": 40001,
  "message": "binding_not_in_project",
  "data": {
    "bindingId": "tb_01",
    "projectId": "prj_02"
  },
  "requestId": "req_xxx"
}
```
- 推荐错误 message 清单
  - invalid_project_id
  - testcase_not_found
  - binding_not_found
  - dataset_not_found
  - api_target_not_found
  - binding_not_in_project
  - run_cannot_be_canceled
  - idempotency_key_payload_mismatch
  并发规则

- 绑定编辑并发控制
  
  - 采用乐观锁字段 version 。
  - PUT 时必须传当前 version ，不匹配返回 40901 + message=version_conflict 。
- 批量执行并发控制
  
  - concurrency 取值建议 1..100 。
  - 超限返回 40001 + message=invalid_concurrency 。
  - 系统可二次限流（项目级并发阈值），触发返回 42901 + message=project_run_rate_limited 。
- 同一用例多绑定并发执行
  
  - 允许；每条 items[] 生成独立 CaseRun 。
  - 去重策略：同一请求内 (testcaseId,bindingId) 不可重复，重复则 40001 + message=duplicate_items_in_request 。
  幂等规则

- 适用范围
  
  - POST /api/runs/from-testcases 、 POST /api/testcases/{id}/bindings 、 POST /api/api-targets
- 键规则
  
  - 头 Idempotency-Key ，最大 128 字符。
  - 服务端按“关键业务字段规范化 JSON”计算签名（与现有 run 实现同思路， run.py ）。
- 行为规则
  
  - 同 key + 同 payload：返回首次创建结果（ code=0 ，同资源 ID）。
  - 同 key + 不同 payload：返回 40901 + message=idempotency_key_payload_mismatch 。
  - 不传 key：按普通创建处理，可能产生重复资源。
- 过期策略
  
  - 建议幂等窗口 24h （可配置）；窗口外 key 可复用。
  与现有接口兼容约束

- 不改变现有 /api/runs 、 /api/runs/{id}/case-runs 的基础字段和状态值， runs.py
- 不改变现有用例 CRUD 请求结构， testcase.py
- 新字段采用“只增不删、可选返回”，避免前端存量页面回归风险
