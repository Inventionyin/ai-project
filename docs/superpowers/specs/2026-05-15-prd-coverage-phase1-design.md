# 智能测试平台 PRD 覆盖式交付设计（Phase 1）

## 1. 文档目的

本文档固化“按 PRD 模块做覆盖式闭环”的一期设计，作为后续实施计划和多 agent 并行开发的基准。

本次设计覆盖第一波子项目：

1. 平台共享底座
2. 需求资产主链路
3. 需求变更影响分析
4. 接口自动化与 TestCase 绑定

目标不是一次性把整个平台做到最深，而是先让核心业务模块全部达到统一验收口径。

## 2. 背景与设计结论

### 2.1 当前底座结论

主代码仓库使用 `D:\OtherProject\ai-project`，继续以现有 `FastAPI + Vue 3 + PostgreSQL + Alembic` 为正式底座，不新建平台工程，不 fork `MeterSphere` 作为第一阶段实现主体。

已具备的正式能力：

- 账号认证、项目管理、用例管理、套件管理、环境管理
- 接口资产 `Collections / Groups / Requests`
- 运行记录 `Runs / CaseRuns / Allure`
- Worker 协议与执行分发
- 需求文档中心、需求分析、测试点审核、用例草稿审核入库、需求用例追溯主链路

### 2.2 为什么采用覆盖式交付

用户已确认采用“先全模块覆盖，再逐步精修”的推进策略，并要求每个模块满足统一的 `C` 级验收标准：

1. 有页面
2. 有 API
3. 有真实数据保存与查询
4. 有一条最小业务闭环
5. 有基础测试

因此，一期不再按零散功能点推进，而是按 PRD 子系统推进闭环。

### 2.3 一期总体决策

采用“方案 A：按 PRD 模块做覆盖式闭环”。

一期交付边界：

- 做强 1 到 4 号子项目
- 让“需求文档 -> AI 分析 -> 测试点 -> 用例草稿 -> 正式用例 -> 接口绑定 -> 执行 -> 回归集合”形成可落地闭环
- 第二波再覆盖线上问题、知识沉淀、治理增强
- 第三波再做外部缺陷系统和 DevOps 插件

## 3. 设计目标

### 3.1 业务目标

把现有平台从“有能力雏形”推进到“可以作为智能测试平台主版本持续开发”的状态。

一期完成后，平台应支持以下主路径：

1. 在项目下上传需求文档并生成版本
2. 对需求版本执行解析和结构化分析
3. 抽取测试点并人工审核
4. 生成用例草稿并批量审核入库
5. 对需求版本差异做影响分析
6. 产出回归用例集合
7. 导入 OpenAPI/Postman
8. 将接口资产与正式 TestCase 建立双向绑定
9. 发起手动执行并查看结果

### 3.2 技术目标

- 复用现有仓库的模型、接口风格、前端页面结构和执行体系
- 所有新增能力继续走 PostgreSQL + Alembic 正式迁移
- 所有新增接口纳入现有鉴权、租户隔离、项目隔离
- 所有页面使用真实后端接口，不保留静态占位
- 新增模块都要有最小 API 测试，关键链路有最小 UI/构建验证

## 4. 非目标

以下内容不在本次一期设计的实现范围内：

- 直接 fork 或兼容 `MeterSphere` 的插件生态
- 完整组织级多租户治理重构
- 复杂 AI 编排平台、模型网关、Prompt 市场
- 完整 Jira / 禅道 / Teambition 双向同步
- 高级报表大屏、复杂 BI 指标中心
- 完整自动告警编排

这些能力进入二期或三期设计，不阻塞一期主闭环。

## 5. 架构原则

### 5.1 后端原则

- 延续 `endpoint -> service -> model/schema` 结构
- 业务规则集中在 `services`
- 接口只做鉴权、参数接收、响应封装
- 新表全部带 `tenant_id`、`project_id`、时间戳
- 统一复用现有 `ApiResponse`、分页和错误模型

### 5.2 前端原则

- 延续 `src/views/*` 页面和 `src/lib/api/*` 调用层拆分
- 优先补正式业务页，不新增无业务价值的装饰页
- 所有状态展示与路由入口接入项目工作台侧边栏
- 页面以工作台风格为主，不引入营销型布局

### 5.3 数据原则

- 以“可追溯”优先于“最短字段”
- 需求、分析、测试点、草稿、正式用例、接口绑定、执行记录之间保留可查询关联
- 版本比较产物落库，不只在前端临时计算

### 5.4 测试原则

- API 新能力以 `pytest` 为基础
- 前端至少通过 `npm build`
- 对高风险主链路增加集成测试
- 所有验收都以真实数据库联调为准，不以假数据视为完成

## 6. 一期范围分解

### 6.1 子项目 1：平台共享底座

#### 6.1.1 目标

补齐 1 到 4 号业务模块共用的最小平台基础设施，避免后续每个模块重复造字段和状态机。

#### 6.1.2 一期范围

- 项目空间入口补齐到正式导航
- 公共枚举与状态字典统一
- AI 任务记录骨架
- 审计日志骨架
- 通用附件/来源信息规范

#### 6.1.3 设计决策

一期不做完整治理中心，而是只补足支撑 1 到 4 号子项目的共用能力：

- `AiJobRecord`
  - 记录分析、测试点同步、草稿生成、影响分析等 AI 或规则任务
  - 字段包括：任务类型、触发人、输入对象、状态、摘要、耗时、错误信息
- `AuditLog`
  - 记录文档发布、分析修订、草稿审核、回归集合生成、接口绑定变更等关键操作
  - 字段包括：模块、动作、对象 ID、操作者、前后摘要、时间
- `AttachmentSource`
  - 统一文档原文件、导入来源、解析来源等字段表达，不单独做文件中心

#### 6.1.4 页面/API/数据/闭环

- 页面
  - 项目设置中新增“平台记录”入口
  - 最小包含 AI 任务记录列表、审计日志列表
- API
  - `GET /projects/{projectId}/platform/ai-jobs`
  - `GET /projects/{projectId}/platform/audit-logs`
- 数据
  - 新增正式表保存记录
- 最小闭环
  - 执行需求分析后，在 AI 任务记录中可看到任务结果
  - 审核草稿入库后，在审计日志中可看到操作留痕

#### 6.1.5 验收标准

- 页面可按时间和模块筛选
- 列表数据来自真实库表
- 至少两类业务动作能产生审计记录
- 至少两类 AI/规则动作能产生任务记录
- 有基础 API 测试

### 6.3 子项目 3：需求变更影响分析

#### 6.3.1 目标

实现 PRD 中“按需求版本识别新增/修改点，输出回归用例集合”的正式模块。

#### 6.3.2 核心业务定义

影响分析不是重新做一次需求分析，而是比较同一需求文档的两个版本，识别：

- 新增需求点
- 修改需求点
- 删除需求点
- 高风险变更点

并基于现有追溯关系输出受影响测试点和回归用例集合。

#### 6.3.3 新增数据对象

- `RequirementChangeSet`
  - 一次版本对比任务
  - 记录基线版本、目标版本、状态、摘要、生成方式
- `RequirementChangeItem`
  - 单条变更点
  - 记录变更类型、主题、原文摘要、新文摘要、风险等级、建议动作
- `RequirementRegressionSet`
  - 一次影响分析输出的回归集合
  - 记录关联变更集、生成说明、统计信息
- `RequirementRegressionCase`
  - 回归集合中的正式用例项
  - 记录测试用例 ID、来源变更点、纳入理由

#### 6.3.4 核心规则

- 一期采用“版本文本差异 + 结构化分析结果比对 + 现有追溯链映射”混合策略
- 不要求一期实现复杂语义 diff 引擎
- 先保证结果可解释、可人工复核

映射顺序：

1. 版本文本分段差异
2. 功能点/规则/测试点名称归一化比对
3. 找到关联 `RequirementTestPoint`
4. 通过 `RequirementCaseLink` 找到正式 `TestCase`
5. 写入回归集合

#### 6.3.5 页面/API/数据/闭环

- 页面
  - 文档详情页新增“变更影响分析”标签
  - 影响分析详情页展示版本对比、变更清单、回归集合
- API
  - `POST /projects/{projectId}/requirements/docs/{docId}/change-sets`
  - `GET /projects/{projectId}/requirements/docs/{docId}/change-sets`
  - `GET /projects/{projectId}/requirements/change-sets/{changeSetId}`
  - `POST /projects/{projectId}/requirements/change-sets/{changeSetId}/regression-set`
  - `GET /projects/{projectId}/requirements/regression-sets/{regressionSetId}`
- 数据
  - 新增 4 张正式表
- 最小闭环
  - 同一文档上传两个版本后可发起对比
  - 对比后可看到变更项
  - 可一键生成回归集合
  - 可跳转到用例详情查看受影响正式用例

#### 6.3.6 验收标准

- 能对同一文档两个版本产生正式落库的变更集
- 回归集合至少包含用例标题、来源变更点、纳入理由
- 可通过真实追溯关系自动选出回归用例
- 有基础 API 测试覆盖对比与回归生成

### 6.4 子项目 4：接口自动化与 TestCase 绑定

#### 6.4.1 目标

把现有接口资产、运行能力和正式测试用例体系连接起来，形成“接口资产 -> 用例绑定 -> 执行 -> 结果回看”的闭环。

#### 6.4.2 已有基础

当前底座已存在：

- `Collections / Groups / Requests`
- OpenAPI / Postman 导入基础
- 运行能力和 `Runs`
- 正式 `TestCase`
- 局部 `testcase bindings` 能力

#### 6.4.3 一期补齐项

- 接口请求与正式 `TestCase` 双向绑定
- 接口集合页正式详情视图
- 绑定视图中显示来源需求与追溯信息
- 手动执行与失败重跑入口串起来
- 最小报告聚合展示

#### 6.4.4 新增或扩展数据对象

- 复用现有 `TestCaseBinding` 或在其基础上扩展
- 绑定对象统一支持：
  - 绑定类型：`REQUEST` / `COLLECTION`
  - 目标对象 ID
  - `testcase_id`
  - 来源：手动绑定 / AI 推荐 / 需求生成
  - 断言摘要
  - 最后执行结果摘要

若现有 `TestCaseBinding` 无法兼容上述结构，则新增 `ApiTestCaseLink`，但优先扩展旧模型，避免双轨模型并存。

#### 6.4.5 正式闭环

1. 导入 OpenAPI 或 Postman
2. 形成接口集合、分组、请求
3. 在请求或集合详情页选择正式 `TestCase`
4. 保存绑定关系
5. 触发手动执行
6. 在运行记录看到结果
7. 在请求详情和用例详情都能看到彼此关系

#### 6.4.6 页面/API/数据

- 页面
  - Collections 列表页接真实数据
  - Collection 详情页
  - Request 编辑/运行面板
  - TestCase 绑定抽屉或侧栏
  - Run 详情页补真实结果
- API
  - 复用现有 collections/runs/testcases 路由
  - 补充绑定查询与写入接口
  - 补充按请求、按集合、按用例三种维度的绑定查询
- 数据
  - 绑定关系正式落库
  - 执行摘要保留最近一次结果快照

#### 6.4.7 验收标准

- 至少支持请求级绑定
- 至少支持单请求执行并回看结果
- 能从接口侧和用例侧双向查看绑定关系
- 至少覆盖导入、绑定、查询、执行记录查看四类测试

## 10. 错误处理与回退策略

### 10.1 通用错误策略

- 对象不存在：返回 404
- 非本项目对象：返回 404，不暴露跨项目信息
- 状态不允许：返回 400
- 重复绑定：返回 409 或幂等成功，具体以现有风格为准
- AI/规则任务失败：返回任务失败状态，并记录错误摘要

### 10.2 业务回退策略

- 草稿批量审核失败时按事务回滚，避免部分入库
- 回归集合生成失败不影响原始变更集
- 绑定保存失败不影响原请求和原用例数据
- 解析失败和分析失败都需要保留失败记录，便于追踪

## 13. 第二波与第三波预告

为避免一期范围膨胀，只记录边界，不展开详细设计。

### 13.1 第二波

- 线上问题录入与分析
- 知识库沉淀
- 角色权限与项目空间 UI
- 审计日志扩展
- 模型策略配置

### 13.2 第三波

- 第三方缺陷系统集成
- DevOps 插件与流水线入口
- 告警机制
- 更完整的治理中心

## 14. 最终结论

一期继续基于现有 `ai-project` 底座，按 PRD 做覆盖式闭环交付，不新起工程，不更换主技术栈。

这份设计的核心是把平台推进到“核心模块都能用，而且每块都是真实落库、真实联调、可测试”的状态。后续实施计划必须严格围绕本设计的一期范围推进，不再扩散到第二波和第三波需求。
