一、范围与边界

- 改动范围（前端）
  - CasesPanel（提交逻辑、状态管理）
  - BatchRunModal（三步交互语义）
  - CaseBinding/DataBinding/Environment 子组件（字段改造）
  - 新增少量 API 调用函数（api-targets/bindings/environments）
- 不改动范围（后端）
  - 保持 /api/api-targets 、 /api/testcases/{id}/bindings 、 /api/runs/from-testcases 现有实现不变
- 目标契约
  - 4.3：绑定配置由 apiTargetId/datasetId/params/version 驱动
  - 4.4：执行请求体由 envId + items[{testcaseId,bindingId}] 驱动
  二、分阶段执行步骤

- Phase 1：请求模型对齐（先打通）
  
  - 把现有提交体从 testcaseIds/environment/datasetNameById/apiNameById 改为 4.4 标准体。
  - 固定补齐 triggerType: MANUAL 、 meta.source: "cases_panel" 。
  - 结果： POST /api/runs/from-testcases 可直接成功消费。
  - 对齐依据： run.py 、 统一方案文档4.4
- Phase 2：Step1 改为“绑定配置选择”
  
  - 当前“选用 API 接口”改为“选择 Binding（每用例必选）”。
  - 每行展示： binding.name + apiTarget摘要 + dataset摘要 ，值是 bindingId 。
  - 保留“一键应用全部”，但只对可兼容用例生效（不可用则给提示）。
  - 结果：生成 selectedBindingByCaseId ，作为 items[] 的核心来源。
- Phase 3：Step2 改为“参数覆盖（可选）”
  
  - 当前 dataset 文本绑定改为 overrideParams 编辑（JSON 可选）。
  - 默认不填，即执行绑定中的原 params 。
  - 结果： items[].overrideParams 仅在用户填写时传递。
- Phase 4：Step3 环境改为真实环境
  
  - 弹窗打开时请求项目环境列表，展示 name/baseUrl ，内部存 envId 。
  - 去掉 development/staging/production 静态枚举作为提交来源。
  - 对齐依据： environments.py
- Phase 5：绑定配置入口补齐（4.3联动）
  
  - 在用例行“绑定配置”列接入真实数据（非 - 占位）。
  - 增加“新建/编辑/删除绑定”操作入口（可用抽屉或小弹窗，不破坏主页面）。
  - 接口： GET/POST /api/testcases/{id}/bindings 、 PUT/DELETE /api/testcase-bindings/{id} 、 GET /api/api-targets 。
- Phase 6：运行结果最小闭环
  
  - 执行成功后拿 runId ，支持跳转运行详情。
  - 下一步可接 GET /api/runs/{runId}/case-runs 展示 bindingSnapshot （可选增强）。
  三、前端任务拆解（可直接分配）

- 任务A：批量执行数据模型重构
  
  - 定义 BatchRunFormState 、 RunFromTestcasesPayload 、 RunItemPayload
  - 新增 adapter： buildRunPayload(state, projectId)
- 任务B：Binding 读取与选择
  
  - 新增 fetchBindingsByTestcaseIds(testcaseIds[])
  - Step1 表格控件把 apiName 改为 bindingId
- 任务C：环境读取与选择
  
  - 新增 fetchProjectEnvironments(projectId)
  - 环境组件 value 从枚举改为 envId
- 任务D：提交前校验
  
  - 必填： envId 、每条用例 bindingId
  - 规则：去重 (testcaseId,bindingId) ； concurrency 在 1..100
  - 错误提示：无绑定、重复项、参数 JSON 非法
- 任务E：绑定配置管理入口
  
  - 用例列表“绑定配置”列接真实统计（如“2个绑定”）
  - 入口支持 CRUD（优先最小可用）
  四、联调验收清单（你可直接拿去评审）

- 请求体验收
  
  - 执行请求必须包含： projectId/envId/triggerType/meta/concurrency/stopOnFailure/items[]
  - items[] 每项包含： testcaseId/bindingId ，可选 overrideParams
- 流程验收
  
  - 用例无绑定时不可执行并有可理解提示
  - 有绑定时可一键执行并返回 runId
  - 幂等 key 重复请求行为符合预期（首次结果复用/冲突提示）
- UI验收
  
  - 三步结构不变、认知成本低
  - 字段文案与业务语义一致（不再出现“API名称文本提交”）
  五、风险与回滚

- 风险
  - 老数据里可能存在无 binding 的历史用例，导致用户“选了用例但不能执行”
- 规避
  - 增加“仅显示可执行用例”筛选；无绑定提供“去配置绑定”快捷入口
- 回滚
  - 保留旧弹窗开关（feature flag），紧急时切回旧流程但禁止线上主用
